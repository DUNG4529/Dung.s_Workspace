# 🧠 Giai đoạn 1: Phân tích bài toán (Decomposition & Pattern Recognition)

> **Mục tiêu:** Ứng dụng Computational Thinking để bóc tách yêu cầu phức tạp thành các phần nhỏ, từ đó định hình cấu trúc dữ liệu và luồng xử lý cho dự án **Employee Payroll Management System** (LAB211).

---

## 1. Phân tích Đối tượng dữ liệu 🔍 (Decomposition - Danh từ)

Đối tượng cốt lõi là **Employee**. Cấu trúc và ràng buộc (Constraints) được trích xuất trực tiếp từ đặc tả:

| Trường dữ liệu (Field) | Kiểu dữ liệu | Ràng buộc & Định dạng (Constraints) |
| :--- | :--- | :--- |
| **Employee ID** | String | 🔹 Không null <br> 🔹 Duy nhất (Unique) <br> 🔹 Regex: `E` + 3 chữ số (VD: `E001`) |
| **Name** | String | 🔹 Không được để trống |
| **Role** | String | 🔹 Chỉ nhận 4 giá trị: `Developer`, `Tester`, `Manager`, `HR` |
| **Base Salary** | Double/Float | 🔹 Phải là số dương (> 0) |
| **Working Days** | Integer | 🔹 Nằm trong khoảng `0` – `26` |
| **Bonus** | Double/Float | 🔹 Phải >= 0 |
| **Status** | String | 🔹 Nhận 2 giá trị: `active` (đang làm) hoặc `inactive` (đã nghỉ) |

> [!info] Lưu ý về lưu trữ
> Hệ thống sử dụng file văn bản **`employees.txt`** làm kho lưu trữ. Dữ liệu được ghi theo định dạng dòng:
> `<id, name, role, baseSalary, workingDays, bonus, status>`

## 2. Phân tích Chức năng ⚙️ (Decomposition - Động từ)

Hệ thống yêu cầu **9 chức năng chính**, được chia thành 2 nhóm hành vi để dễ thiết kế kiến trúc:

### 🗂️ Nhóm 1: Quản lý tập hợp danh sách
- **1. Load from file:** Đọc dữ liệu từ `employees.txt`.
- **2. Add employee:** Nhập mới, kiểm tra hợp lệ (validate) và thêm nhân viên.
- **3. Update employee:** Cập nhật Role, Base Salary, Bonus hoặc Status.
- **4. Remove employee:** Xóa nhân viên theo ID.
- **5. Search employee:** Tìm kiếm theo ID, Name, Role hoặc Status.
- **7. Display Employee List:** Hiển thị toàn bộ danh sách.

### 💾 Nhóm 2: Tương tác với thiết bị lưu trữ & Điều khiển
- **6. Calculate payroll:** Tính tổng lương cho nhân viên `active`.
- **8. Save to file:** Ghi dữ liệu đã cập nhật xuống file.
- **9. Quit program:** Thoát chương trình (có xác nhận lưu file trước khi thoát).

## 3. Nhận diện khuôn mẫu (Pattern Recognition) 🧩

Khi quan sát các chức năng, đặc biệt là `Add` và `Update`, bạn sẽ thấy thao tác **Nhập liệu + Kiểm tra hợp lệ (Validation)** xuất hiện lặp đi lặp lại.

> [!danger] Vấn đề phát sinh
> Nếu sử dụng các khối `if...else` hoặc `try...catch` rải rác ở mọi nơi, code sẽ bị:
> - **Dư thừa** (Code duplication)
> - **Khó bảo trì** (Low maintainability)
> - **Vi phạm** nguyên tắc *Low Coupling - High Cohesion*

### ✅ Giải pháp Pattern Recognition:

Thay vì kiểm tra thủ công, hãy nhận diện **mẫu dữ liệu (Data Pattern)** và sử dụng **Regular Expression (Regex)**.

> [!tip] Chiến lược triển khai tối ưu
> 1. Khai báo tất cả các chuỗi Regex (định dạng ID, giới hạn số, tập hợp Role) thành **hằng số** trong một Interface (ví dụ: `Acceptable`).
> 2. Tạo lớp `Inputter` (Utility class) chứa các phương thức nhập chung.
> 3. **Lợi ích:** Tập trung logic kiểm soát dữ liệu vào một nơi (Single Responsibility), tái sử dụng (Reusable) dễ dàng và code ngắn gọn hơn.

```java
// Ví dụ minh họa về Interface chuẩn hóa
public interface Acceptable {
    String ID_REGEX = "^E\\d{3}$";
    String ROLE_REGEX = "^(Developer|Tester|Manager|HR)$";
    String STATUS_REGEX = "^(active|inactive)$";
    // ... các ràng buộc số khác
}
```

## 💡 GỢI Ý CORE PROMPT CHO AI AUDIT LOG (Decision-Making)

Việc lựa chọn cách xử lý Validation là một **quyết định kiến trúc (Architectural Decision)** quan trọng. Bạn có thể sử dụng AI để phân tích và ghi lại quá trình ra quyết định như sau:

> [!example] Prompt mẫu để hỏi AI
> *"Compare using Regex vs Try-Catch blocks for validating constraints like ID format (E+3 digits), Role enumeration, and positive Salary in a Java console application. Which approach provides better cohesion and reusability for my Employee Payroll project?"*

### 📝 Human Delta & Reflection (Ví dụ phản hồi của bạn):

> [!note] Quyết định cuối cùng
> Sau khi tham khảo AI, tôi quyết định chọn **kết hợp Regex + Interface `Acceptable` và lớp `Inputter` dùng chung**.
>
> **Lý do:** Regex rút ngắn đáng kể số dòng code (LOCs), giảm thiểu xử lý `try-catch` gây ảnh hưởng hiệu suất trong vòng lặp nhập. Cách này đảm bảo các chức năng `Add` và `Update` đạt chuẩn **Low Coupling** và **High Cohesion** theo đúng kiến trúc môn LAB211.
